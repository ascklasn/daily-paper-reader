## 摘要
生物医学智能体有望自动化复杂的生物工作流程，然而当前系统面临两个根本瓶颈：生物信息学工具在接口和执行环境上高度异构，而智能体规划仍依赖于扁平的提示检索工具描述。随着生物医学生态系统的增长，这种工具覆盖范围与上下文大小之间的耦合导致工具混淆、规划不稳定以及执行效率低下。我们提出BioManus，一个基于MCP原生的生物医学智能体，建立在结构化生物能力之上的图支撑规划。BioManus首先引入BioinfoMCP编译器，将异构的生物信息学软件转换为标准化的MCP服务器，从而形成一个大型可执行的MCP生态系统。然后，它将这个生态系统组织成一个类型化的异质MCP图，涵盖工具、操作、数据类型和工作流阶段。在推理时，BioManus检索紧凑的任务特定子图，合成操作级工作流支架。这种设计将规划复杂度与原始工具库存大小解耦，在高召回率检索下实现了上下文压缩比Θ(N / (h * m_bar))，其中N是工具总数，h是工作流长度，m_bar（远小于N）是每个操作的平均候选工具数。在BioAgentBench和LAB-Bench上的实验表明，与先进的生物医学智能体基线相比，BioManus在执行准确性、工作流有效性和上下文效率方面均有提升。这项工作提出了一个范式转变：可扩展的生物医学推理需要结构化的可执行能力图，而不是日益增大的提示级工具检索。

## Abstract
Biomedical agents promise to automate complex biological workflows, yet current systems face two fundamental bottlenecks: bioinformatics tools are highly heterogeneous in interfaces and execution environments, while agent planning still relies on flat prompt-retrieved tool descriptions. As biomedical software ecosystems grow, this coupling between tool coverage and context size leads to tool confusion, unstable planning, and inefficient execution. We introduce BioManus, an MCP-native biomedical agent built on graph-scaffolded planning over structured biological capabilities. BioManus first introduces the BioinfoMCP Compiler, which converts heterogeneous bioinformatics software into standardized MCP servers, yielding a large executable MCP ecosystem. It then organizes this ecosystem as a typed heterogeneous MCP graph over tools, operations, datatypes, and workflow stages. At inference time, BioManus retrieves compact task-specific subgraphs, synthesizes operation-level workflow scaffolds. This design decouples planning complexity from raw tool inventory size, achieving a context compression ratio of Theta(N / (h * m_bar)) under high-recall retrieval, where N is the total tool count, h is the workflow horizon, and m_bar (much smaller than N) is the average number of candidate tools per operation. Experiments on BioAgentBench and LAB-Bench show that BioManus improves execution accuracy, workflow validity, and context efficiency over advanced biomedical agent baselines. This work suggests a paradigm shift: scalable biomedical reasoning requires structured executable capability graphs rather than increasingly larger prompt-level tool retrieval.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：生物医学领域存在大量异构工具（CLI、Python、R、容器化流程等），当前LLM驱动的生物医学代理（如Biomni）普遍采用基于提示的工具检索范式——将检索到的工具描述直接注入到LLM上下文中进行规划。这种范式在工具规模较小时有效，但随着生物信息学工具生态急剧增长（N持续变大），检索窗口需同步增大，导致上下文稀释、工具混淆、规划不稳定和执行效率下降。
- **核心问题**：现有方法将规划复杂度与工具库存大小强耦合（`C_prompt = O(k * l_t)`），且忽略了生物工作流本身的结构化特性（数据类型兼容性、阶段顺序等）。生物医学工具并非存在于扁平语义空间，而是构成隐式的可执行图结构。
- **整体含义**：论文主张范式转变——从**基于提示的扁平工具检索**转向**基于结构化能力图的推理**。只有显式建模工具、操作、数据类型和工作流阶段之间的图关系，才能实现可扩展的、稳定的规划。

## 2. 方法论：核心思想与关键技术细节

### 核心思想
构建一个**类型化的异质MCP图**（Typed Capability Graph），将工具生态组织为节点（工具、操作、数据类型、工作流阶段、MCP服务器、能力）和边（实现、消费/产生、托管、阶段转换等）。推理时，不暴露全部工具，而是检索紧凑的任务相关子图，在操作级别进行结构化图推理，再绑定到具体MCP工具。这样规划复杂度从`O(N)`降至`O(h * m_bar)`。

### 关键技术组件
1. **BioinfoMCP编译器**：
   - 三阶段流水线：准备（从文档、帮助信息提取规范）、执行（LLM生成MCP服务器Python代码，自动解析验证，失败重试）、交付（打包依赖和Docker环境）。
   - 生成910个MCP服务器，3500个可调用工具，覆盖8个生物领域。
2. **类型化能力图构建**：
   - 六类节点：工具（V_T）、操作（V_O）、数据类型（V_D）、能力（V_C）、阶段（V_S）、MCP服务器（V_R）。
   - 边类型：`implements`、`consumes/produces`、`hosts`、`follows`（软工作流边缘）等。
   - 自动构建：LLM标注器从工具模式推断输入/输出数据类型、语义操作类别、阶段注释。
3. **图支撑工作流规划**：
   - 输入查询`q` → 结构化任务规范 → GraphRAG检索获得任务特定子图及候选操作路径提示。
   - LLM规划器合成操作级执行路径`P = (o1, o2, ..., oh)`，每个操作关联紧凑候选工具集`T(oi)`。
   - 只动态注册选中的MCP服务器，其余不加载。
4. **复杂度分析**：
   - 提示式规划：`C_prompt = O(k * l_t)`，其中k≈N（高召回时）。
   - 图支撑规划：`C_graph = O(h * (l_o + m_bar * l_s))`，与N无关，只与工作流长度h和每操作候选工具数m_bar有关。
   - 压缩比理论：`Θ(N / (h * m_bar))`，当N很大时优势显著。

## 6. 主要结论与发现

1. **提示式检索在小型工具空间中有效，但随着规模扩大迅速退化**：从无MCP到2000工具，几何平均令牌消耗增加4.2倍，方差变大，规划不稳定。

3. **图规划比提示检索更节约上下文**：BioManus的上下文消耗稳定在约1.3M令牌级别（LAB-Bench），而Biomni-2k消耗2.62M。

5. **消融实验证实两个组件均有贡献**：MCP基础设施带来3.0~8.9个百分点的提升，图规划在此基础上再提升4.2~4.7个百分点。
6. **范式转变**：可扩展的生物医学推理需要结构化执行能力图，而非更大的提示级工具检索。

## 7. 优点

- **方法创新性**：将工具生态组织为类型化图，引入操作级检索和规划，从根本上解耦工具规模与规划复杂度，理论压缩比清晰。
- **系统完整性**：编译器自动将异构工具标准化，使得系统可以直接执行真实生物信息学工具（910个MCP服务器），实用性强。
- **实验设计严谨**：同时评估端到端工作流和知识问答；对上下文消耗进行定量分析；消融实验隔离每个组件的贡献；编译器跨模型验证。
- **开放透明**：提供完整的提示模板、图统计、案例研究、工具目录，可重复性高。
- **基准表现优秀**：在多个任务上超越所有基线，特别是在需要多步骤推理的SeqQA和CloningScenarios上优势明显。

## 8. 不足与局限

- **文档依赖**：编译器依赖现有工具文档（手册、帮助信息）生成接口，文档质量直接影响转换可靠性。缺乏执行级验证可能漏掉细节。
- **图注释精度有限**：当前使用轻量语义注释，可扩展为更细粒度生物本体和专家审核的工作流知识。
- **检索质量依赖**：图检索和操作规划的质量影响最终效果，如果子图不完整或规划出错，可能不如全检索。
- **静态图更新问题**：图构建后需跟上新工具发布，持续维护成本未知。
- **未在更大规模生态（如10^4+工具）上验证**：最高测试2000工具，但理论压缩比应在更大规模更显著。

- **伦理与可解释性**：作为辅助系统，专家监督仍必要，论文未讨论生物结论的可靠性或潜在风险。
