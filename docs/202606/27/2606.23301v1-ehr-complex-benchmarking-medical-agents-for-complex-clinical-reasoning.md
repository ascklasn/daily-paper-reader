## 摘要
临床智能体有望使电子健康档案（EHR）的获取民主化，然而现有基准未能反映实际EHR分析的复杂性，例如通常通过静态SQL生成在理想化、干净的EHR上操作，而非交互式执行。在本工作中，我们提出了EHR-Complex，一个专为交互式临床数据库推理设计的大规模基准。基于大型MIMIC-IV数据平台（36.5万患者、31张表、5亿+记录），EHR-Complex包含约5.2万个任务，涵盖六种临床意图，支持患者级和人群级查询，每个任务要求智能体通过执行SQL查询或Python代码与沙盒环境交互。值得注意的是，EHR-Complex考虑了真实世界中SQL任务的复杂性，涉及纵向多表聚合与组合推理，平均每个查询包含31.93个SQL结构组件。在EHR-Complex上的评估结果揭示了这些EHR推理场景的临床难度，表现最好的模型仅达到62.3%的精确匹配准确率。几乎所有评估模型在k=4时的Pass^k一致性均低于50%，暴露出广泛的随机脆弱性。对代表性LLM的超过3800条失败轨迹进行细粒度分析，揭示了三种主要失败模式：SQL逻辑错误、医学代码查找失败和语义误解。EHR-Complex为临床智能体提供了一个严格的试验平台，并突显了大规模EHR分析中鲁棒推理方面仍存在的差距。

## Abstract
Clinical agents promise to democratize access to electronic health records (EHRs), yet existing benchmarks fail to reflect the complexity of practical EHR analysis, e.g., often operating on idealized, clean EHRs via static SQL generation rather than interactive execution. In this work, we introduce EHR-Complex, a large-scale benchmark designed for interactive clinical database reasoning. Built on the large MIMIC-IV substrate (365K patients, 31 tables, 500M+ records), EHR-Complex comprises about 52K tasks spanning six clinical intents, supporting both patient-level and population-level queries, where each task requires an agent to interact with a sandboxed environment by executing SQL queries or Python code. Notably, EHR-Complex considers the real-world SQL task complexity for longitudinal multi-table aggregation and compositional reasoning, resulting in 31.93 SQL structural components per query on average. Evaluation results on EHR-Complex reveal the clinical difficulty of these EHR reasoning scenarios, with the top-performing model achieving only 62.3% exact-match accuracy. Pass^k consistency drops below 50% for nearly all evaluated models at k=4, exposing broad stochastic fragility. A fine-grained analysis of more than 3,800 failed trajectories for representative LLMs reveals three dominant failure modes: SQL logic errors, medical-code lookup failures, and semantic misunderstandings. EHR-Complex provides a rigorous testbed for clinical agents and highlights remaining gaps in robust reasoning for large-scale EHR analysis.

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：构建一个大规模、交互式的临床数据库推理基准EHR-Complex，要求智能体在沙盒环境中通过执行SQL或Python代码与环境交互，从而得出最终答案。
- **数据构建管道**（三步）：
  1. **患者事件图（Patient Event Graph）**：基于MIMIC-IV全量数据，为每位患者构建异构时序图，将入院、ICU停留、实验室事件、诊断、处方等标准化为事件节点，并用时间/语义关系连接。
  2. **临床证据路径（Clinical Evidence Path）**：从事件图中采样时间有序、临床合理的证据路径，作为候选查询的推理链。
  3. **证据到SQL模板**：使用LLM将证据路径编译为可执行的SQL程序，通过数据库执行验证（必须成功执行、返回非空结果、可提取确定性答案）。最终得到796个经过验证的SQL模板。

- **复杂度指标**：平均每个查询包含31.93个SQL结构组件（嵌套、聚合、连接、条件、分组/排序、窗口函数），远超现有基准的3.06~17.03。

## 6. 主要结论与发现

1. **整体难度高**：性能最好的模型（Kimi-K2.5和Qwen3.5-397B）仅达到约62.3%精确匹配准确率，表明当前LLM在复杂临床推理上仍有很大提升空间。
2. **人群级查询显著更困难**：性能从患者级（~0.83~0.85）降至人群级（~0.40~0.41），尤其是诊断意图中，Kimi-K2.5从0.85降至0.29。错误在逐患者对齐、代码查找或过滤阶段就会悄然改变队列。
3. **鲁棒性差**：Pass^k一致性随k增加快速下降（k=4时多数模型低于50%），且Pass@k（采样多次取最佳）提升有限，说明单次运行不稳定。
4. **三大失败模式**：SQL逻辑错误（Kimi-K2.5中23.1%，GPT-4.1中41.1%）、医学代码查找失败（分别28.6%、32.1%）、语义误解（分别37.9%、14.4%）。
5. **SFT有效**：使用成功轨迹微调后，Qwen3-14B从0.30提升至0.45，Qwen3-32B从0.36提升至0.55，超越GPT-4.1并接近Qwen3-235B，证明轨迹监督能够提供可学习的程序性监督。

## 7. 优点

- **真实性**：基于完整的MIMIC-IV数据库，覆盖31张表和5亿+记录，任务复杂度显著高于现存的任何EHR基准（平均31.93 SQL组件 vs 现有最高17.03）。

- **大规模与多样性**：约5.2万个任务，覆盖6种临床意图和2种范围，支持细粒度分析。
- **鲁棒性分析全面**：引入Pass^k和Pass@k等多种评估维度，揭示模型的随机脆弱性。
- **失败模式细粒度分析**：对3800+失败轨迹进行人工+自动标注，给出清晰的错误分类，为后续改进提供方向。

- **未评估成本与延迟**：未报告不同模型在推理时间、API调用成本或计算资源消耗上的差异。
