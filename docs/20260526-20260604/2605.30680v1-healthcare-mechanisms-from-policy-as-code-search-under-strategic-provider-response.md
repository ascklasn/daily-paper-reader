## 摘要
医疗机制与其引发的策略性提供者响应密不可分：现有的医疗AI基准固定了这种响应，因此无法通过其产生的均衡来评估机制。我们将医院机制设计重新定义为语言模型的程序合成：可类型化、可检查的规则程序由Medi-Sim执行和评分，Medi-Sim是一个包含五个策略性提供者渠道（编码、选择、延迟、努力、分诊）的多智能体模拟器。激励扫描将经典的健康经济学发现恢复为相邻制度——在利润压力下的向上编码和低复杂度患者选择，以及古德哈特式漂移，其中测量绩效与真实结果反相关——并且单个审计杠杆暴露了压力迁移：关闭编码渠道会使低复杂度选择增加一倍以上。然后，LLM引导的进化代码搜索在相同的规则程序空间内合成了一个可检查的混合目标程序，该程序消除了向上编码，将拒绝率减半，并保留了大部分利润导向基线的资金。

## Abstract
Healthcare mechanisms are inseparable from the strategic provider response they induce: existing healthcare AI benchmarks hold this response fixed and so cannot evaluate mechanisms by the equilibrium they produce. We recast hospital mechanism design as program synthesis for language models: typed, inspectable rule programs are executed and scored by Medi-Sim, a multi-agent simulator with five strategic provider channels (coding, selection, delay, effort, triage). An incentive sweep recovers classical health-economics findings as adjacent regimes -- up-coding and low-complexity-patient selection under profit pressure, and Goodhart-style drift where measured performance becomes anti-correlated with true outcomes -- and a single audit lever exposes pressure migration: closing the coding channel more than doubles low-complexity selection. LLM-guided evolutionary code search over the same rule-program space then synthesizes an inspectable mixed-objective program that eliminates up-coding, halves rejection, and retains most of the profit-oriented baseline's funds.

- **核心问题**：现有的医疗AI基准在评估医院机制（如支付规则、审计、奖金池）时，**固定了提供者（医生/医院）的策略性响应行为**，因此无法捕捉机制引发的实际均衡效果。例如，一套新支付规则可能诱导医生通过“上调编码”（up-coding）或“挑选低复杂度患者”来最大化自身利益，而基准测试只能测评规则文本本身。
- **总体含义**：医疗机制的本质是**行政指令与提供者策略性响应**的复合产物。作者认为，机制设计应被视为一个**闭合循环**：规则→提供者行动→再反馈到规则评估。现有工作要么忽略提供者响应（如临床RL），要么使用黑盒神经网络（不可审计），不适用于高监管的医疗场景。
- **驱动现象**：文献中已广泛报道的“压力迁移”（pressure migration）——关闭一个漏洞（如加强审计编码）会导致提供者将压力转移到相邻渠道（如选择性拒绝病人）。这要求评估必须同时追踪所有渠道，而非单一指标。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- 将医院机制设计转化为**程序合成**（program synthesis）任务：管理员规则写成**类型化、可检查的DSL程序**，然后在**多智能体模拟器Medi-Sim**中执行并评分，最后使用**LLM引导的进化搜索**（AlphaEvolve）优化该程序。
- 关键前提：所有规则必须是**逐行可审计**的，因此排斥神经网络控制器；LLM仅作为**代码编辑算子**，而非策略主体。

### 关键技术细节
- **Medi-Sim 环境**：实现了一个**Identify–Produce–Settle（IPS）循环**。
    - **Identify**：患者到达，携带真实诊断组、复杂度（CMI）、紧急性等，医院编码员可选择报告不同组（coding wedge）。
    - **Produce**：医疗团队（服务单元）基于队列、激励、疲劳等做出**五通道动作**：编码（coding）、选择（selection）、延迟（delay）、努力（effort）、分诊（triage/resource）。
    - **Settle**：结算报销、计算KPI、分配奖金池（通过softmax锦标赛）。
- **提供者响应类（Π_P）**：采用**有界理性**参数化行为规则，而非求解完全均衡。每个通道根据局部梯度调整（如编码通道使用候选组得分函数，包含审计惩罚、道德压力等）。
- **管理员策略类（Π_A）**：一个**领域特定语言（DSL）**，包含固定杠杆集：激励系数(α, β)、总容量、柔性容量、奖金池、KPI权重、审计强度等。所有编辑限定在这些字段上。
- **LLM引导的进化搜索**：
    - 使用AlphaEvolve算法（基于OpenEvolve实现）：候选程序经语法/类型检查后，在短期和完整随机批次上评估。
    - 适应度函数 = 期望回报 − λ_unsafe × 安全/扭曲惩罚 − λ_var × 方差惩罚（式4）。
    - LLM（ChatGPT-5.4）作为结构化的突变算子，对程序代码进行语义编辑。

### 公式/算法流程（文字说明）
- **Stackelberg目标**：管理员（领导者）优化某个社会目标o（福利、利润或混合），考虑提供者（追随者）的有界理性最佳响应。最终适应度 = 折扣期望回报 − 安全惩罚 − 方差正则项。
- **模拟器循环（算法A.1）**：每周期：管理员设杠杆 → 患者到达 → 编码员报告 → 路由/排队 → 各团队动作 → 容量约束下治疗 → 结算/审计 → 更新状态。
- **搜索循环（算法A.2）**：初始化策略库 → 迭代K次：部署候选程序 → 在验证种子模拟 → 计算评分 → 保留/变异 → 最终在测试种子评估。

## 6. 论文的主要结论与发现

1. **经典失败模式构成一个连续相图**：L1扫描显示，低激励区域限制准入，利润压力激化编码和选择，质量压力增加努力和预算压力，而平衡区域则将压力转向延迟和KPI代理博弈。不同经典现象（Dafny式上调编码、Bevan目标博弈）映射到相图的不同区域。
2. **压力迁移是结构性障碍**：L2表明，关闭一个渠道（如加强审计减少编码）会导致提供者转向其他渠道（如提高选择拒绝率）。奖金池扩大恶化KPI-健康对齐；柔性容量在KPI导向下反而增加等待。
3. **LLM引导的代码搜索能重塑挣得回报的渠道**：在混合目标下，搜索得到的策略消除了上调编码，将拒绝率减半，并保留大部分利润导向基线的资金。关键机制是降低了本地奖金压力，而非提升总分。
4. **搜索依赖多样化热身库**：只有包含多样性（利润/质量/中性等）的热身库才能被搜索有效改进；中性单独库时搜索失败。因此程序搜索在此场景中是“结构化精炼工具”而非零样本生成器。

## 7. 优点：方法或实验设计上的亮点

- **创新性问题建模**：将医疗机制设计转化为程序合成，兼顾可审计性与策略性响应，填补了现有研究的空白（既不是临床RL也不是纯机制设计）。
- **压力迁移的显式诊断**：IPS分解和五通道状态使得可以追踪压力在各个渠道间的重新分配，而不仅仅看总分数。这比单一指标评估更贴近现实。
- **可解释性**：最终策略是**代码**，可逐行阅读和合规审查，符合医疗设备监管要求（GMLP指南）。
- **三层实验验证**：从现象复制（L1）到杠杆效应（L2）再到优化搜索（L3），层层递进，证据链清晰。
- **消融实验严谨**：热身库消融、K=300诊断等都揭示了搜索成功的关键因素，不回避局限性。
- **较低的算力需求（推测）**：相比训练大规模神经网络，主要使用模拟和LLM推理，资源门槛较低。

## 8. 不足与局限：包括实验覆盖、偏差风险、应用限制

- **完全合成环境**：模拟器是机械性的，非校准临床部署模型。参数设置（如到达率、效用系数、审计策略）均假设固定；真实医院的复杂度、编码空间、患者行为等远更复杂。
- **有界理性假设**：提供者响应使用参数化行为规则而非均衡求解，可能过度简化了策略性博弈。实际中提供者可能形成更复杂的适应性策略，甚至集体行为（如共谋）。
- **搜索结果依赖热身库**：当前L3混合策略效果来自对特定热身库的改良；若初始库不包含有利指向，搜索无法自行发现。这限制了方法在其他场景下的通用性。
